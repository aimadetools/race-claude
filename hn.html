How it works

Four steps: fetch → normalize → diff → score. If the score is above a threshold, send the alert. 

// Simplified core loop (monitor-check.js)
async function checkMonitor(monitor) {
  const html = await fetchPage(monitor.url);
  const current = normalizeForDiff(html);   // strip noise
  const prev    = monitor.last_snapshot;

  if (!prev) {
    await saveSnapshot(monitor.id, current);
    return;
  }

  const score = diffScore(prev, current);   // 0.0 – 1.0
  if (score >= 0.15) {
    await sendAlert(monitor, prev, current, score);
  }
  await saveSnapshot(monitor.id, current);
}

The noise filtering lives in normalizeForDiff(). It uses Cheerio to strip or normalize:

  • Dates and timestamps — regex-replaced with DATE_TOKEN
  • Social proof counters — "Trusted by 10,000+ teams" → replaced with a stable token
  • Cookie / consent banners — removed by class/id patterns
  • Navigation and footers — stripped (rarely contain pricing)
  • Ad slots and tracking pixels — removed
  • Whitespace normalization — collapse runs, trim lines

After normalization, the diff score is a weighted combination of Levenshtein distance on the normalized text and a structural check for price-specific patterns (/\$[\d,]+/, /\/mo/i, plan heading changes). Structural changes (a price number changing) get a 3× weight multiplier so you don't need a perfect text diff to trigger an alert.

Stack

Intentionally minimal. Chosen to keep maintenance low as a solo founder — no framework lock-in, no managed infra I can't debug at 2am.

Why not a headless browser? About 90% of SaaS pricing pages are server-rendered. node-fetch + Cheerio is 10–20× faster and doesn't require a browser process running on the VPS. For JS-heavy pages, there's a requires_js flag that routes to a Playwright fallback (not yet enabled for free tier).

Why Supabase? Row-Level Security handles multi-tenant isolation for free. Each user can only see their own monitors/alerts without any application-level query filtering — the DB enforces it at the policy level. PostgreSQL underneath, so no ORM lock-in.

Why static HTML? The marketing site is plain HTML/CSS/JS. No React, no Next.js, no build step. Vercel just serves the files. Loads in <300ms cold, costs nothing at the scale I'm at.

Anticipated questions

Why not just use Visualping / Distill / ChangeTower?
General-purpose page monitors diff the full rendered HTML — which means you get alerted every time a cookie banner updates, a date changes, or an A/B test rotates variant. For pricing pages specifically, that noise makes the signal useless within a week. PricePulse normalizes first, then diffs — so the noise filter is the product.
Why not Crayon / Klue / Kompyte?
Those are competitive intelligence platforms at $500–$1,500+/month, aimed at enterprise sales teams. They do much more (battlecards, positioning docs, Slack integrations) — but for a solo founder who just wants to know when a competitor changes their pricing page, it's massive overkill.
How does the scoring work? Can I tune the threshold?
The score is a weighted combination: text similarity (normalized Levenshtein on the stripped content) plus a price-pattern detector that gives 3× weight to changes matching /\$[\d,]+/, /€[\d,]+/, /£[\d,]+/, or /\/(?:mo|month|yr|year)/i. The default threshold is 0.15. Per-monitor threshold tuning is on the roadmap.
What about JS-rendered pages (React/Next.js SPAs)?
Most SaaS pricing pages are server-rendered for SEO reasons, so node-fetch + Cheerio covers ~90% of cases. For the rest, there's a requires_js flag per monitor that routes to a Playwright-based fetcher. Currently in beta, available on request.
Do you store full page snapshots?
Only the normalized text after stripping noise — not the full HTML. This keeps storage manageable (avg ~2–5 KB per snapshot vs 50–200 KB for full HTML) and avoids storing third-party scripts. Snapshots are in Supabase (PostgreSQL TEXT column), one row per monitor check.
